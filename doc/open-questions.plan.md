## 一、业务层问题

### Q-001: 是否需要多用户支持?

- **类别**: 业务边界
- **状态**: 待确认
- **描述**: 当前应用为单用户桌面应用，所有配置和会话存储在本地用户目录 (`~/.starxo/`)。是否有场景需要支持多用户共享同一远程服务器的沙箱环境? 例如团队协作场景下，多人连接同一台开发服务器的不同容器。
- **影响**: 若需要多用户支持，需要引入用户认证、会话隔离、容器资源配额等机制，架构变化较大。
- **当前假设**: 单用户使用，无需多用户支持。

### Q-002: 是否需要本地沙箱 (非 SSH)?

- **类别**: 业务边界
- **状态**: 待确认
- **描述**: 当前必须通过 SSH 连接远程服务器才能使用。是否需要支持本地 Docker Desktop 作为沙箱环境? 这将降低使用门槛，开发者无需准备远程服务器即可体验。
- **影响**: 需要新增本地 Docker 适配层，与远程 SSH+Docker 模式并存。`sandbox` 包需要抽象出通用接口。
- **当前假设**: 仅支持远程 SSH+Docker 模式。

### Q-003: 是否需要 Agent 自主学习能力?

- **类别**: 产品方向
- **状态**: 待确认
- **描述**: 当前 Agent 不具备跨会话学习能力，每次对话基于当前消息历史推理。是否需要支持 Agent 记忆用户偏好、项目上下文、常用命令等? 例如记住用户偏好的编程语言、代码风格、项目结构。
- **影响**: 需要引入长期记忆存储 (向量数据库/文件)、RAG 检索、提示词注入等机制。
- **当前假设**: 无跨会话学习能力，每次对话独立。

## 二、实现层问题

### Q-004: 检查点存储是否需要从内存切换到持久化?

- **类别**: 架构决策
- **状态**: 待确认
- **描述**: 当前使用 `store.NewInMemoryStore()` 作为 Eino ADK 的 CheckPointStore。这意味着 Agent 的中断/恢复状态在应用重启后丢失。若用户在中断等待状态下关闭应用，再次启动后无法恢复之前的执行位置。
- **当前行为**:
  - 中断状态 (`PendingInterrupt`) 存储在内存中
  - 应用重启后挂起的中断无法恢复
  - 正常会话消息历史通过 `SessionStore` 持久化，不受影响
- **可选方案**:
  - 方案 A: 保持内存存储 (简单，中断场景不频繁)
  - 方案 B: 实现基于文件的 CheckPointStore (可恢复，增加复杂度)
  - 方案 C: 在应用关闭时自动取消挂起的中断 (折中)
- **影响**: 方案 B 需要实现 `compose.CheckPointStore` 接口的文件版本，序列化 Agent 状态。
- **当前假设**: 使用内存存储，中断状态不跨重启持久化。

### Q-005: 上下文窗口策略是否需要更智能的摘要替代截断?

- **类别**: 功能增强
- **状态**: 待确认
- **描述**: 当前窗口化策略 (`WindowMessages`) 对超出窗口的旧消息使用简单的数量丢弃和内容截断。丢弃的消息仅留下 `"[Earlier conversation with N messages omitted for brevity]"` 占位符。是否需要更智能的摘要策略?
- **当前行为**:
  - 保留最近 20 条消息 (含第一条系统消息)
  - 单条消息超过 4000 字符时截断 (保留前 60% + 后 20%)
  - 丢弃的消息用数量占位符替代
- **可选方案**:
  - 方案 A: 保持当前截断策略 (简单高效)
  - 方案 B: 使用 LLM 生成丢弃消息的摘要 (准确但增加 API 开销)
  - 方案 C: 使用关键信息提取 (工具调用结果、代码修改等) 生成结构化摘要
- **影响**: 方案 B/C 增加 LLM 调用次数和延迟，但可能提升长对话的上下文质量。
- **当前假设**: 使用简单截断策略。

## 三、当前假设

### A-001: 文档基于现有代码生成

- **描述**: 本文档体系中的所有技术描述、接口定义、业务规则均基于当前代码库 (commit `68cb548`) 的静态分析生成。随着代码迭代，文档需要同步更新。
- **风险**: 代码变更后文档可能滞后。
- **缓解**: 在 M2 测试基础设施阶段，考虑引入文档一致性检查工具。

### A-002: 以内部逻辑为核心

- **描述**: 文档以代码的内部实现逻辑为核心，而非外部行为规格。这意味着文档描述的是"代码如何工作"而非"产品应该如何工作"。没有独立的产品需求文档 (PRD)。
- **风险**: 实现细节可能偏离理想的产品行为。
- **缓解**: 在 Q-001 ~ Q-003 确认后，可补充产品需求文档。
